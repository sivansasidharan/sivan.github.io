Overview

 Synapse is a lightweight and high-performance Enterprise Service Bus (ESB), based on simple XML and acts as a Web services management and integration broker. Synapse provides an approach to SOA that can expose existing applications without rewriting any code. With exceptional support for XML, Web Services, REST and several other content interchange formats, such as plain text, binary, Hessian and JSON, Synapse becomes an attractive choice and a key approach to an SOA strategy.

ESB Concept

An ESB is a standards-based integration platform that combines messaging, web services, data transformation, and intelligent routing to reliably connect and coordinate the interaction of significant numbers of diverse applications across extended enterprises with transactional integrity.

Business events in an event-driven enterprise can affect the normal course of a business process in any order and at any time. Applications that exchange data in automated business processes need to communicate with each other using an event-driven SOA to have the agility to react to changing business requirements. In an ESB, applications and event-driven services are tied together in an SOA in a loosely coupled fashion, which allows them to operate independently from one another while still providing value to a broader business function.

An ESB provides the implementation backbone for an SOA. The ESB model promotes SOA-based integration with service contracts and metadata associated with the service contracts, such as Web Services Description Language (WSDL), schema, or policy and it provides a loosely coupled, event-driven SOA.

The ESB model ensures that every message that goes into the bus ends up in the right place.


Apache Synapse Architecture Principles & Deployment Models
 
The principle on which Synapse is built is as below:
  • SOA model
 Synapse is based around messages that move around a network, messages that could be of any format.
Synapse which is built around XML and Web services standards basically molds them into standard information and takes into account the SOAP idea of an envelope containing header and body.

Synapse also helps in ensuring security by using Web services standards, such as WS-ReliableMessaging and WS-Security thus ensuring reliable connections between applications. These types of quality-of-service (QoS) capabilities are basically achieved without modifying existing applications. For example, by placing a Synapse instance directly in front of an existing XML/HTTP service, you can add support for WS-Security. This can add support for digital signatures, which can be important in a B2B environment.
  • HTTP and SOAP – Interaction Models
Synapse utilizes HTTP in two different ways – as a transport protocol and more over as an interaction protocol that tends to follow the REST model. When used in such a way it tends to be strict about the semantics of the HTTP verbs (GET, PUT, POST, and DELE). HTTP as the normal transport protocol works in the normal XML request/response pattern. While with SOAP, the universal, platform neutral, transport-independent messaging format, it's being used for many applications and it’s also the basis for many higher-level specifications. Used together or separately, SOAP and HTTP form a universal interaction model and can connect heterogeneous systems across and within enterprise boundaries.

Synapse at its core uses the SOAP model. So the internal message formats implementations and data structures reveal the use of a SOAP envelope pattern. So as message flow is redirected to the synapse engine it’s been transformed to and from various formats including REST, SOAP, JavaScript Object Notation (JSON), and JMS, with standard mediators. In general, Synapse supports many different transports, including Java™ Message Service (JMS), Mail, XML/HTTP, and SOAP.
  •  XML – Universal data model
XML has grown as the universal data format that enables mixt applications to pass data in a common data model and that’s readily understood by any language. The Synapse world is built entirely with XML in mind. Since all interactions are using XML, the entire network is converted to a bus while these XML messages are pushed as far as possible in your network and forms the ESB, while synapse would restrict its duties of managing this entire bus.

 Synapse deployment models

Synapse can be deployed in a few different ways. The simplest way to deploy Synapse is using the concept of a proxy.
  •     Proxy model
In the proxy model, you explicitly define a new endpoint (proxy) that Synapse listens on. It poses that a new service is been written, except that the service logic is implemented by a remote service. In the simplest instantiation of this model, you directly wire the proxy listener to a remote endpoint. Even in this simple case you can do some useful things like, switch between XML/HTTP and SOAP and JMS, turn on or off protocols, such as WS-Addressing, WS-ReliableMessaging, and WS-Security.

A Sample request/response routing through Synapse

A Sample Synapse configuration XML


<definitions xmlns="http://synapse.apache.org/ns/2010/04/configuraiton">


    <proxy name="GetCustomerInfoProxy">


        <target>


            <endpoint>


                <address uri="http://localhost:9000/services/CustomerInfoService"/>


            </endpoint>


            <outSequence>


                <send/>


            </outSequence>


        </target>


        <publishWSDL uri="file:sample/sample_proxy_1.wsdl"/>


    </proxy>


</definitions>

The above configuration will expose a proxy service named GetCustomerInfoProxyand specifies an endpoint (http://localhost:9000/services/CustomerInfoService)as the target for the proxy service.
Any incoming messages to the proxy service (http://localhost:8280/services/GetCustomerInfoProxy) will be directed to the address http://localhost:9000/services/CustomerInfoServicespecified in the endpoint. There is also an out sequence for the proxy service, which is applicable for response messages. In the out sequence, we just send the messages back to the client. The publishWSDLtag specifies a WSDL to be published for this proxy service. 
  •  Policy-Based Deployment
In this approach, you configure your clients to send all messages to Synapse (Synapse acting as an HTTP proxy). Synapse has a default sequence of actions that applies to any messages that arrive in this way. Techniques like content-based filters (xPath or Regex) can then be used to selectively apply policies to particular messages.

Mediation Engine


Apache Synapse implements a generic mediation framework. We can specify a sequence of mediations that operates on a message — either based on which endpoint it arrived through or on some filter. Synapse provides built-in support for mediations like:

  • Logging to log4j.

  • Modifying headers, including WS-Addressing headers.

  • Running scripts, including JavaScript, Groovy, Ruby, and others supported by Bean Scripting Framework (BSF). 
    • E4X support allows JavaScript to directly manipulate XML messages without needing to use APIs like DOM. 
    • Similar support exists in Ruby as well.

  • Enabling and terminating WS-Security and WS-ReliableMessaging.

  • Switching between SOAP versions and XML/HTTP.

  • XML Schema validation.

  • Faulting messages or dropping them.

*E4X - ECMAScript for XML is a programming language extension that adds native XML support to ECMAScript (which includes ActionScript, DMDScript, JavaScript, and JScript).

*MTOM - MTOM is the W3C Message Transmission Optimization Mechanism, a method of efficiently sending binary data to and from Web services.


Connect, manage, and transform Enterprise services


Synapse can connect, manage, and mediate our services.

Connect - Synapse can act as a gateway to our enterprise and manage the exposure of services to partners more effectively. The gateway model can include applying authentication and authorization to incoming messages (using the Apache Rampart framework to support WS-Security and WS-SecureConversation). By mapping virtual URLs to real URLs inside our network, Synapse enables our enterprise to maintain a uniform addressing scheme to the outside world while hiding the internal infrastructure details.
Synapse can be used to map simple GET HTTP requests into XML/JMS connections to a back end. In this way, legacy resources can be exposed to a much wider range of clients, including Asynchronous JavaScript + XML (Ajax) applications and mobile devices.
Manage - Synapse provides a consistent model to administer policy and other aspects of services. By interposing Synapse as an intermediary, you can control aspects like authentication, auditing, logging, or QoS aspects like failover and load balancing. All these features can be enabled by configuring a simple XML file without writing any code.(eg. All requests to http://abcbank.com/personal can be authenticated, logged, and randomly logged and marked for auditing.)
Transform - Synapse can transform your legacy applications into state-of-the-art Web services. Synapse can act as a proxy and virtualize the service and application by doing one or more of the following:
Performing an XSL Transformation (XSLT) to the SOAP message (headers or body)
Having JavaScript/E4X transformers map XML directly into JavaScript
Using a plain old Java object (POJO), where you can write Java logic to manipulate the message
Mapping JSON to XML, where a JSON/HTTP stream can be sent out as a SOAP message, or vice versa
Performing attachment mediation, where conversions like SOAP with Attachments to Message Transmission
Optimization Mechanism (MTOM), Base64 to MTOM, and vice versa are supported.
Performing your own custom transformation
Mediation inside Synapse is designed with care, and there are two possible types of mediations that can happen inside Synapse. They are:

  • Message Mediation - Managing and transforming the messages flowing between the client and a service in an enterprise.
  • Service Mediation - Mediating messages coming into a specific service by specifying the target URI as a Synapse mediation service.
In message mediation, the client does not know about the mediation of the message in the middle of transmission. When it comes to service mediation, the client directly speaks to Synapse and specifies the address of the service to which the message should go for mediation. Inside these services, it is not required to check the transport headers and take the routing decision because the messages are already filtered. Whereas in message mediation, most of the time Synapse will have to check the transport headers to take the routing decision. These services are called Proxy Services in Synapse.
With Synapse at the edges, we can transform our entire network into an ESB, which provides a centralized and consistent way of managing our service offerings.
Extending Apache Synapse ESB

Synapse provides a number of extension points to plug-in custom developed code there by extending the functionality of ESB. This can include - 
Writing custom mediators: The primary interface of the Synapse API is the MessageContext interface. Mediators can read and/or modify the message encapsulated in the MessageContext in any suitable manner - adjusting the routing headers or changing the message body.

Synapse Observers: This can be developed using SynapseObserver interface or by extending the AbstractSynapseObserver class. A Synapse observer is notified by the Synapse configuration when new elements are added to the configuration and when existing elements are removed from the configuration. It can be used to log any received events.

Scheduled Tasks: This could be custom developed using Java and could be scheduled in the ESB to execute periodically. The Synapse Task Interface should be used for the scheduler which has an execute ‘method’ and once scheduled the execute method is called by Synapse periodically. 

Summary

Synapse provides a simple, easy-to-use open-source alternative to rebrand your existing middleware as an SOA platform with minimum cost and effort. Synapse maximizes reuse by mediating or transforming existing services without coding, and it provides a good model to integrate your services into an ESB model.
